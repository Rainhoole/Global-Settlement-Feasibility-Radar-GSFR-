# GSFR V1 开发复盘（2026-03-01）

沉淀本次 V1 开发与上线过程中的有效经验、问题根因和可复用流程，为后续 V2 开发降低返工和沟通成本。

## 2. 本次已完成事项（结果）

- 完成项目初始化提交并推送到 GitHub。
- 配置 GitHub Pages 自动构建发布（Vite + Actions）。
- 修复站点空白页问题并完成线上可访问验证。
- 将当前稳定版本标记为 `V1`（`version=1.0.0`，tag=`v1.0.0`）。
- 补充项目主文档 `README.md`（包含 V1/V2 版本策略）。

## 3. 关键问题与根因

### 3.1 空白页问题（最核心）

现象：
- 页面标题可显示，但页面主体空白。
- 控制台出现 `404` 资源错误（例如 `/src/main.tsx` 或错误路径的 `assets`）。

根因：
- GitHub Pages 子路径部署（`/Global-Settlement-Feasibility-Radar-GSFR-/`）未正确设置 Vite `base`。
- Pages 曾一度直接发布源码 `index.html`，导致生产环境仍引用开发入口 `/src/main.tsx`。
- CDN/浏览器缓存导致修复后短时间仍可能看到旧页面。

修复方案：
- 新增 `build:github`：`vite build --base=/Global-Settlement-Feasibility-Radar-GSFR-/`。
- 新增 `.github/workflows/deploy-pages.yml`，统一由 CI 构建 `dist` 并发布。
- 通过带参数 URL（cache busting）验证新版本是否已生效。

结论：
- 子路径部署项目必须在构建层显式配置 `base`。
- “能本地跑”不代表“能 Pages 跑”。

### 3.2 数据改了但页面没变

现象：
- 修改 `docs/*.csv` 后，线上 UI 没有变化。

根因：
- 前端运行时读取的是 `src/data/countries.json`，不是 `docs/*.csv`。
- 目前 `docs` 与 `src/data` 之间没有自动同步链路。

修复方案：
- 直接修改 `src/data/countries.json` 并重新部署。
- 在 README 中明确数据来源与同步关系。

结论：
- 先定位“运行时数据源”再改数据，避免改错位置。

### 3.3 一次错误替换导致国家中文名错位

现象：
- 期望修改瑞典/挪威，但误影响了丹麦条目（顺序替换导致偏移）。

根因：
- 使用“就近文本替换”而非“按 code 锚点定位替换”。

修复方案：
- 按 `code=DK/SE/NO` 块逐段校验后再改。
- 修复为：`DK=??`、`SE=瑞典`、`NO=挪威`。

结论：
- 对结构化数据应采用“键值锚定修改”，并做最小范围回归检查。

## 4. 本次有效做法（建议保留）

- 每次改完都执行本地构建：`npm run build`。
- 对线上问题先看控制台错误与网络请求，再改代码。
- 改动后立即检查 GitHub Actions 状态，确认部署是否真正完成。
- 用版本标签管理里程碑：`v1.0.0` 作为可回溯基线。

## 5. V2 开发建议流程（可直接执行）

1. 从 V1 标签创建分支：
   - `git checkout -b v2-dev v1.0.0`
2. 所有功能在 `v2-dev` 迭代，主分支只接收稳定变更。
3. 每个功能 PR 至少包含：
   - 变更说明（What/Why）
   - 影响范围
   - 本地构建或测试结果
4. 发布前检查清单：
   - 构建成功
   - 关键路径手测（地图、筛选、搜索、详情）
   - Pages 线上验证
5. V2 发布时打标签：
   - `v2.0.0`

## 6. 针对 V2 的重点改进项

- 建立 `docs/*.csv -> src/data/countries.json` 自动转换脚本（消除手工同步）。
- 统一文本编码（UTF-8）并清理历史乱码字段。
- 增加数据完整性校验（例如 `name_zh` 为空/占位符检测）。
- 增加基础自动化测试（至少覆盖数据加载和关键筛选逻辑）。
- 增加 `CHANGELOG.md`，记录每次发布变化。

## 7. 可复用命令清单

```bash
# 本地开发
npm run dev

# 本地构建
npm run build

# GitHub Pages 构建（子路径）
npm run build:github

# 查看工作流运行
# GitHub 页面: Actions -> Deploy GitHub Pages

# 基于 V1 开始 V2
git checkout -b v2-dev v1.0.0
```

V1 已形成可上线、可回滚、可继续迭代的稳定基线。  
进入 V2 前，优先补齐“数据自动同步”和“编码一致性”两条工程能力，可显著减少后续维护成本和线上偏差风险。
